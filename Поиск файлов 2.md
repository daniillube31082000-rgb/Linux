В первой части мы разобрались с тяжёлой артиллерией – командой find, которая ищет файлы по сложным критериям, но при этом может работать не очень быстро, если вы обшариваете всю систему. Во второй части мы рассмотрим инструменты, которые дают быстрый результат, но с определёнными условиями.

### **1. locate**
``locate`` не является базовой в том смысле, что она предустановлена во всех Linux-дистрибутивах "из коробки". Она поставляется **вместе с пакетом** ``mlocate`` (или его аналогами, такими как ``slocate``), который нужно установить вручную в некоторых системах.

+ **Зачем нужен locate?**
+ **Это быстрая утилита** для поиска по **предварительно собранной** базе файлов.
+ В отличие от ``find``, не обходит файловую систему в реальном времени, поэтому выдаёт результат за долю секунды.
+ **Пример использования**:  

    ``locate document.pdf``

                  
Он тут же покажет все пути, где обнаружено ``document.pdf``, если файл на момент создания базы существовал в системе.  

Пример вывода:

``/home/user/Documents/document.pdf``  

``/home/user/Downloads/document.pdf``  

``/var/www/html/uploads/document.pdf``

                  
+ **Обновление базы:**
+ База обычно обновляется по расписанию (``cron``), либо вручную через ``sudo updatedb. *updatedb`` тоже входит в состав ``mlocate``.
+ Если вы создали новый файл, а база не обновилась, ``locate`` **может его не знать**.
У ``locate`` есть несколько полезных флагов:

+ ``-i`` — игнорировать регистр (например, найдёт ``File.txt`` по запросу ``file.txt``);

+ ``-n <число>`` — ограничить количество результатов;

+ ``-r <регулярное выражение>`` — использовать регулярные выражения для поиска.

⚠️ Обратите внимание: опции вроде ``-exec`` или фильтрация по дате относятся к ``find``, а не к ``locate``.

### **2. which**
+ **Поиск исполняемых файлов**, т. е. программ, которые можно запустить без указания полного пути.
+ **Как это работает?**
+ Утилита смотрит переменную окружения ``$PATH``, где перечислены директории с командами (например, ``/usr/bin``, ``/usr/local/bin``).
+ Если вы набираете в терминале ``python``, ``which`` показывает, где лежит исполняемый файл ``python``.
+ **Пример:**  
``which python``  

    ``which ls``

                  
Результат: полный путь до бинарного файла, например ``/usr/bin/python``.
+ **Зачем нужно?**  

Чтобы понять, какую именно версию программы вызываете, если в системе есть несколько одноимённых бинарников.
### **3. whereis**
Ищет бинарные файлы, исходники и руководства (``man-страницы``), связанные с конкретной командой.  

+ **Пример:**  

    ``whereis gcc``

                  
Показывает, где лежит исполняемый файл ``gcc``, где находятся его исходники (если установлены) и файлы справки (``man``).
+ **В чём разница с which?**
+ ``which`` ориентируется только на ``$PATH`` и отвечает, где лежит исполняемый файл.
+ ``whereis`` даёт более широкий набор мест — бинарники, исходный код, ``man-страницы``.
Чтобы посмотреть, в каких директориях ``whereis`` ведёт поиск, используйте команду:

    ``whereis -l``

                  
Она выведет список каталогов, которые сканируются по умолчанию.

### **Итог**
+ ``locate`` — моментально находит файлы по имени, но опирается на базу, которая обновляется не в реальном времени.
+ ``which`` — показывает путь к исполняемой команде (программе) в ``$PATH``.
+ ``whereis`` — ищет файлы исходников и ``man-страницы``, помимо самого бинарника.  

Если вы ищете любой файл в текущем режиме здесь и сейчас и нужны сложные условия (время изменения, размер, права), используйте ``find``. Если вам просто нужно быстро узнать, где лежит программа или упоминаемая ``document.pdf``, вероятно ``locate`` или ``which``(если это исполняемый файл) будут самым удобным выбором.
